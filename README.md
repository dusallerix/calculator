Учебный проект, помогающий освоить базу JavaScript. 
Несмотря на простейшую задумку, помогает разобраться в теме создание структуры и логики,обработка инпута и ошибок, а также работа с CSS и flex.

## Основные функции

Калькулятор предназначен для ускорения математических вычислений, является подручным инструментом если нужно что-то быстро посчитать. 
Именно поэтому, он должен отвечать базовым стандартам: ввод, работа с данными и их вывод, а также их изменение/удаление.

Что должно быть:
1. Ввод с кнопок калькулятора
2. Вывод на экране данных: полного мат.выражения и последнего введенного числа/оператора
3. Подсчет математических функций и вывод результата.
4. Удаление: полное(всех данных) и частичное (последнего введенного)
5. Тригонометрические функции (sin, cos, tan)
6. Другие мат.выражения: степень, корень, факториал
7. Поддержка скобок(т.е. приоритет решения)
  
## Алгоритм сортировочной станции, или же алгоритм преобразования в ОПЗ (обратную постфиксную запись)  

В связи с ТЗ, калькулятор должен был поддерживать длинные математические выражения с разнообразными функциями и поддержкой скобок. Это уже давало намеки на то, что прибегать к eval() будет глупо и бессмысленно. Он использовался лишь в начале, дабы проверить обработку инпута и прочие вещи, но не логику. И вот, когда работа дошла до последних трех пунктов свыше, было принято решение создать отдельный класс Math.     
        
Тут сразу возникал проблема - как можно было разбить длинное выражение и считать его? Записывать каждое выражение и высчитывать его отдельно было бы долго и нелогично, учитывая особенность в приоритете операций.
Разобравшись в том, как работают технические калькуляторы, был найден [Алгоритм сортировочной станции](https://ru.wikipedia.org/wiki/%D0%90%D0%BB%D0%B3%D0%BE%D1%80%D0%B8%D1%82%D0%BC_%D1%81%D0%BE%D1%80%D1%82%D0%B8%D1%80%D0%BE%D0%B2%D0%BE%D1%87%D0%BD%D0%BE%D0%B9_%D1%81%D1%82%D0%B0%D0%BD%D1%86%D0%B8%D0%B8).        
        
Алгоритм сортировочной станции (Shunting Yard) служит для преобразования математических выражений из привычной всем инфиксной записи (где операторы стоят между операндами, как в '1 + 2 * 3') в постфиксную (обратную польскую нотацию, '1 2 3 * +'), удобную для дальнейшего вычисления.

После преобразование записи в постфиксную, начинается её счтение и вычисление. ОПЗ считывает по одному элемента массива, закидывая его стек и при наличии оператора с числом/ми (ведь для разных мат.выражений требуется или один, или два оператора. к примеру, '2^3' требует два числа, а для '5!' требует одно число). 
  
## Возникшие проблемы

1. **Бесконечные операторы**
  
На сырой стадии написания кода, когда только была реализована базовая функция ввода, надо было уже начать ограничивать инпут. Конечно можно было закрыть эту всю тему простым алертом,
но если есть возможность грамотно построить ввод и чтобы пользователь не получал бесконечные ошибки, то надо делать. Решается это все обычным делом:
созданием временном переменной, которая содержит последний символ и его проверкой.

```javascript
let lastChar = ex.slice(-1);

// если последний символ является оператором
if (/[\+\-\*\/]/.test(lastChar)) {
  ...
}
```

|Дано|Ввод|Результат|Примечание|
|-|-|-|-|
|61|/|61/|Последний символ не является оператором, так что ввод числа заканчивается и обрывается оператором '/'|
|*|-|-|Оператор '*' заменился на минус|
|+|+|+|Остается без изменений, предотвращение повторения одинаковых операторов|

    
2. **Минус и его экслюзивность**

Этот пункт уже исходил из первого. В математике, минус является скорее исключением из правил из-за отрицательных чисел, так что он может стоять вместе с другим оператором. 
Поэтому, приходится добавлять больше условий к первому пункту.

'./script.js':
```javascript 
//определяем начало ввода, является ли оно пустое
if (!ex) {
  //если введенный оператор является минусом, то добавляем
  if (op === '-') {
    ...
  }
  return; //иначе прерываем ввод
}
```

|Дано|Ввод|Результат|Примечание|
|-|-|-|-|
| |-|-|Минус стоит в начале выражения|
|-|-|-|Прошлый код не допускает повторения одинаковых операторов|
    
Его экслюзивность также проявляется внутри класса MathLib, который отвечает за расчет выражения. В этом случае, унарный минус будет иметь приоритет не меньше приоритета умножения и деления. 
Для различия унарного минуса от обычного, он будет обозначаться как '~'.    

3. **Ограничение по символам**
  
Для удобства, пришлось добавить функцию на проверку длины вводимого числа. Теперь числа больше 14 цифр не принимаются и калькулятор попросит пользователя стереть последнее веденное число. 
В итоге, имеется три способа удаления: посимвольное (button '->'), по последнему вводному (CE - Clear Entry) и полное удаление (AC - All Clear).

|Дано|Ввод|Результат|Примечание|
|-|-|-|-|
| |100000000000000|'NUMBER TOO LONG'|Попросит пользователя стереть данное число с помощью частичного удаления, при этом не сбрасывает всё выражение, которые было до этой ошибки|
